💥 Buffer Overflow Fundamentals
§1

A buffer overflow occurs when a program writes more data into a memory region than it was allocated to hold. Languages such as C and C++ expose raw memory through pointer arithmetic and provide no inherent bounds enforcement — making unsafe writes invisible at runtime unless explicit instrumentation is present.

Classic stack-based exploitation begins with a fixed-size buffer allocated on the stack inside a called function. When unchecked input arrives via APIs such as gets() or strcpy(), excess bytes overflow the buffer and overwrite adjacent stack metadata — including the saved frame pointer and the return address. By controlling the return address, an attacker redirects execution to injected shellcode or reusable code sequences already present in the binary.

Figure 1 — Stack frame before and after a buffer overflow. 0x41 = ASCII 'A'. Canary corruption triggers *** stack smashing detected *** before the corrupted return address executes.
The OS cannot prevent intra-process memory corruption. OS protections isolate processes from each other — not from their own logic errors. All buffer overflow exploitation happens entirely within the victim process's address space.
🛡 Standard Mitigations & Their Limits
§2

No single mitigation eliminates memory corruption. Each defense addresses a specific attack vector while leaving others exposed. Understanding the limits of each control is as important as deploying it.

MitigationWhat It ProtectsBypass / Limit
Stack CanariesReturn address overwrite on stackDoes not protect heap metadata. Bypassable with information leak of canary value.
Non-Executable Memory (NX/DEP)Code injection via stack/heap shellcodeDoes not prevent ROP (Return-Oriented Programming) — attacker reuses existing code.
ASLRPredictable addresses for gadgets, libc, heapEntropy is limited. Bypassable via format string leaks, side channels, or heap spraying.
Control Flow Integrity (CFI)Indirect control-flow hijackingOverhead and deployment complexity. Does not protect data-only attacks.

Defense in depth is the only realistic strategy. Layer canaries, NX, ASLR, CFI, sanitizers in development, and bounds checking together. No single mechanism is sufficient.
📏 Approaches to Bounds Checking
§3

Guard Pages (Electric Fence)

Unmapped pages placed adjacent to allocations cause an immediate segfault on overflow. Highly effective for debugging — every overflow is caught at the exact offending instruction. Too memory-intensive for production: each allocation requires a full page of guard space.

Fat Pointers

Bounds metadata (base, size) is stored inside the pointer itself, enabling exact per-pointer bounds checking at every dereference. Requires ABI changes — fat pointers are wider than regular pointers, breaking interoperability with unmodified libraries and system interfaces. Performance overhead on every pointer operation.

Shadow Metadata (Baggy Bounds, ASan)

Bounds information is stored in a separate, compact side table indexed by memory address or slot number. Avoids ABI changes — existing pointer widths are preserved. Checking uses constant-time table lookups. Granularity is coarser than fat pointers (slot-sized, not byte-exact), introducing some internal fragmentation.

⚡ Baggy Bounds — Design & Tradeoffs
§4

Baggy Bounds (Akritidis et al., USENIX Security 2009) rounds every allocation up to the nearest power of two and aligns the block accordingly. A compact side table stores the logarithm of the block size for each slot — one byte per slot, enabling O(1) lookup via bit shifts and masks.

Figure 2 — Baggy Bounds slot table. Each allocation is rounded to the next power of 2. The side table stores log₂(size) per slot (1 byte). Bounds checking is three bitwise operations — no metadata traversal required.

Key Tradeoffs

ABI compatibility✅ Unchanged❌ Incompatible✅ Unchanged
Check cost✅ O(1) bitwise⚠️ Per-deref⚠️ Shadow lookup
Granularity⚠️ Power-of-2 slot✅ Exact byte✅ 8-byte aligned

Production-safe✅ Yes❌ Overhead too high❌ 2–3× slowdown
🗺 Process Memory Layout
§5
Figure 3 — x86-64 Linux virtual address space. Addresses from the GDB info proc mappings output in §12. ASLR randomizes the gaps between segments at each exec().

When malloc() is called, the allocator selects a strategy based on size. Small allocations come from thread-local caches (tcache in glibc) or size-class bins. Large allocations use mmap(). Each allocated chunk stores metadata in the bytes immediately preceding the returned user pointer — making adjacent overflows directly dangerous to allocator state.

After realloc(), the original pointer is invalidated — even if the address did not change. Any code retaining the old pointer commits use-after-free.
⚙️ Allocator Internals
§7

glibc — ptmalloc2

Each chunk has a header storing size and state flags in the 8–16 bytes preceding the user data pointer. Free chunks are stored in doubly-linked bins sorted by size. Historically, corrupted list pointers in free chunks enabled "unlink attacks" — arbitrary writes during the coalescing operation. Modern glibc adds pointer integrity checks, but metadata corruption remains a powerful primitive when those checks can be bypassed.

glibc chunk layout (64-bit)
/*   8 bytes  */ prev_size   // size of previous chunk (if free)
/*   8 bytes  */ size        // size of THIS chunk | flags (P=prev_in_use, M=mmap, A=arena)
/* user data  */ <returned pointer starts here>
/* (if free)  */ fd, bk      // forward/backward links in free bin
/* (if free)  */ fd_nextsize, bk_nextsize  // for large bin skip lists

jemalloc

Separates metadata from user data via extents and runs. Uses multiple arenas for concurrency, fine-grained size classes, and thread caches. Profile-friendly: built-in allocation profiling via MALLOC_CONF=prof:true. Less vulnerable to traditional heap overflows because metadata is not adjacent to user data.

Hardened Allocators (Scudo, dlmalloc-hardened)

Add integrity-protected chunk headers (checksummed), randomized allocation ordering, and stack canary-like values inside chunk metadata. Dramatically reduce exploitability of heap corruption — attackers cannot reliably predict or forge the protected header values without a separate leak.

🔬 Debugging & Inspection Techniques
§9
AddressSanitizer — build and run
# Compile with ASan instrumentation
gcc -fsanitize=address -fsanitize=undefined -g -O1 -o target target.c

# Run — ASan intercepts the process and reports:
#   ==PID==ERROR: AddressSanitizer: heap-buffer-overflow on address 0x…
#   READ of size 4 at 0x… thread T0
#       #0 0x… in vulnerable_function target.c:42
./target
Valgrind — memory error detection
# Full error detection with call stacks
valgrind --leak-check=full --show-leak-kinds=all \
         --track-origins=yes --verbose ./target

# Track file descriptors too
valgrind --track-fds=yes ./target
GDB — heap inspection workflow
# Break on malloc/free to trace allocations
break malloc
break free
commands 1
  silent
  printf "malloc(%ld) = %p\n", $rdi, $rax
  continue
end

# Watchpoint — halt when specific memory is written
watch *0x555555559010   # chunk header address

# pwndbg — heap state visualization
heap           # list all chunks
bins           # show tcache/fastbins/smallbins
vis_heap_chunks # color-coded heap map
arena          # main_arena struct

# Inspect chunk header manually
x/4gx 0x555555559000   # prev_size, size, fd, bk
✅ Heap Corruption Investigation Checklist
§10
Work smallest-to-largest: minimize the test case first, then apply tools in order of overhead. Never modify source data during investigation — always work from a reproducible copy.
  1. Minimize the reproducing test case — smallest input that reliably triggers the crash
  2. Run under AddressSanitizer — catches most heap/stack issues with precise attribution
  3. Run under Valgrind if ASan unavailable or doesn't catch it — slower but more exhaustive
  4. Break on malloc/free in GDB — build a timeline of all allocations and frees
  5. Set watchpoints on chunk headers — catch the exact write that corrupts metadata
  6. Inspect tcache lists for duplicates, impossible pointers, or out-of-range values
  7. Examine free list bins — coalescing bugs show up as corrupted fd/bk pointers
  8. Run a fuzzer (AFL++, libFuzzer) against the component — surface hidden paths
  9. Goal: identify the corrupting instruction, not just the downstream crash site
🩺 Troubleshooting Guide — Common Scenarios
§11

Scenario 1 — Crash Inside free()

Symptom: *** Error in 'a.out': free(): invalid pointer *** or double free or corruption

Workflow
break free
run
# When free() is hit, inspect the pointer in $rdi:
info registers rdi
x/4gx ($rdi - 16)     # examine chunk header: prev_size, size flags
# If size field has garbage: a previous write corrupted this chunk's header
# Set watchpoint on the header to find who wrote it:
watch *($rdi - 8)
run                   # re-run; GDB will stop at the corrupting instruction

Scenario 2 — Stack Buffer Overflow (our GDB session case)

Symptom: *** stack smashing detected ***: terminated + SIGABRT

Workflow
break vulnerable_function   # break before overflow
run
info frame                  # show saved RIP, saved RBP
x/64x $rsp                  # dump stack — look for 0x41414141 fill pattern
# Note where 0x41 bytes stop: offset from RSP = required padding to reach return address
watch *(char*)($rbp + 8)    # watchpoint on return address slot
continue

Scenario 3 — Use-After-Free

ASan report + GDB workflow
# Step 1: ASan identifies UAF immediately with allocation/free/access stacks
gcc -fsanitize=address -g -O0 -o target target.c && ./target
# ==PID==ERROR: AddressSanitizer: heap-use-after-free
# READ of size 8 at 0x... — this is the stale pointer access
# freed by thread T0 here: — this is the free()
# previously allocated by thread T0 here: — this is the malloc()

# Step 2: In GDB without ASan, poison freed memory to catch stale pointers
break free
commands
  memset($rdi, 0xde, <size>)   # fill freed region with 0xde = dead sentinel
  continue
end
# Crash will now occur much closer to the stale access

Scenario 4 — Tcache Poisoning / Double Free

pwndbg workflow
# After suspected double free:
tcache_entries   # pwndbg: list all tcache bins and their entries
# Look for: same address appearing twice in the same bin
# Look for: next pointer pointing outside heap range
# Look for: count field mismatching actual number of entries

break __libc_malloc
commands
  printf "malloc(%ld) returning %p\n", $rdi, $rax
  continue
end
# If malloc returns 0x0 or an invalid address: tcache is poisoned

Scenario 5 — Crash After realloc()

Track old pointer invalidation
break realloc
run
# Save the old pointer before call:
set $old_ptr = $rdi
# After realloc returns ($rax = new pointer):
finish
set $new_ptr = $rax
# Set watchpoint on old pointer to catch any post-realloc access:
watch -l *$old_ptr
continue
# Any access to old_ptr after this point is a use-after-realloc bug
🖥 Annotated GDB Session — Live Buffer Overflow
§12
This is a real GDB session on Ubuntu 22.04 debugging a compiled buffer_overflow binary. The binary was compiled without debug symbols (stripped). Annotations explain what each output reveals.
1. Load binary and execute
$ gdb ./buffer_overflow
Reading symbols from ./buffer_overflow...
(No debugging symbols found in ./buffer_overflow)
↑ Binary is stripped — no function names or line numbers available.
  This limits GDB visibility but we can still inspect registers, stack, and memory.

(gdb) run
Starting program: /root/buffer_overflow
[Thread debugging using libthread_db enabled]
2. Overflow detected — SIGABRT triggered
AAAAAAAAAA…AAAAAAAAA  ← 700+ bytes of 'A' (0x41) fed to stdin
*** stack smashing detected ***: terminated
↑ Stack canary was corrupted. glibc's __stack_chk_fail() raised SIGABRT
  before the corrupted return address could execute.
  This is the canary protection working as designed.

Program received signal SIGABRT, Aborted.
__pthread_kill_implementation (...) at ./nptl/pthread_kill.c:44
↑ We are paused inside the SIGABRT handler — deep in pthread/libc internals.
  The actual overflow site is much earlier in the call stack.
3. Inspect process memory map
(gdb) info proc mappings

0x555555554000–0x555555555000  r--p  ← ELF headers (read-only)
0x555555555000–0x555555556000  r-xp  ← .text: executable code
0x555555557000–0x555555558000  r--p  ← .rodata: read-only data
0x555555558000–0x555555559000  rw-p  ← .data / .bss: writable globals
0x555555559000–0x55555557a000  rw-p  ← [heap] ~132KB, grows upward
0x7ffff7c00000–0x7ffff7dbd000  r-xp  ← libc.so.6 text (no write, no exec for data pages)
0x7ffff7fc1000–0x7ffff7fc3000  r-xp  ← [vdso]: kernel-provided fast syscalls
0x7ffffffde000–0x7ffffffff000  rw-p  ← [stack]: ~132KB, grows downward
0xffffffffff600000             --xp  ← [vsyscall]: legacy compat page

Note: NX is active — no segment is both writable AND executable.
      This prevents direct shellcode injection into the stack or heap.
4. Registers at SIGABRT time
(gdb) info registers
rax  0x0       ← syscall return value (0 = pending)
rdi  0x1740f   ← thread ID (pthread_kill arg)
rsi  0x1740f   ← also thread ID
rdx  0x6       ← signal number 6 = SIGABRT
rbp  0x1740f   ← CORRUPTED: RBP should be a stack address (0x7fff…), not 0x1740f
               ↑ We are deep in libc signal handling — this is not the overflow frame's RBP
rsp  0x7fffffffdf60   ← current stack pointer (in signal handler frame)
rip  0x7ffff7c969fc   ← executing inside __GI___pthread_kill+300 (libc)
r12  0x6       ← signal number preserved
5. Stack memory dump — finding the 0x41 fill
(gdb) x/64x $rsp
↑ Dump 64 dwords starting at current stack pointer

0x7fffffffe2f0 → 0x7fffffffe560:  ← region filled with 'A' bytes
0x41414141  0x41414141  0x41414141  0x41414141  ← AAAA AAAA AAAA AAAA
0x41414141  0x41414141  ...                       ← 700+ bytes of attacker input
...
0x7fffffffe570: 0x00414141 0x00007fff ← fill ends here (null terminator from input)
                             ↑ 0x00007fff = environment pointer — not part of overflow

The overflow started at 0x7fffffffe2f0 (buf[]) and grew upward.
Offset from RSP to the 0x41 region = return address + saved RBP were overwritten
before the canary check fired.

Beyond the fill: ASCII-encoded environment variables (PATH, SHELL, TERM…)
visible as readable strings at higher stack addresses — useful for ROP gadget leaks.
6. Breakpoint on main and stack frame inspection
(gdb) break main
Breakpoint 3 at 0x555555555219   ← main+8 (entry point after prologue)

(gdb) info breakpoints
Num  Type        Enb  Address              What
1    breakpoint  y    <PENDING>  point     ← function "point" not found (stripped binary)
3    breakpoint  y    0x555555555219       <main+8>

With stripped binaries:
  - Use: break *0x<address>  instead of break function_name
  - Use: disassemble $pc, +100  to read instructions around crash
  - Use: x/20i 0x555555555219  to examine main's disassembly
